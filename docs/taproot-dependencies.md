# Taproot integration

Waystone supports `@gnolith/taproot >=0.3.0 <0.5.0` as an optional peer. It is optional because Waystone consumes browser-safe structural JSON and does not import Taproot at runtime; a generated Site using Taproot must provide a compatible release. Its adapters use Waystone-owned structural types for the canonical JSON fields they consume. The emitted declarations do not import or re-export Taproot, so an ordinary UI import does not resolve Taproot's Diamond, RDF, or persistence declaration graph. Waystone remains free of D1, Diamond, Drizzle, and Cloudflare runtime dependencies. The wider range reflects that browser protocol boundary, not assembled Site certification.

Taproot uses canonical Wikibase JSON while Waystone uses a UI-oriented display model. The root export provides explicit adapters for stored entity envelopes, claims, snaks, references, search pages, and revisions. `createWaystoneClient()` applies those adapters to canonical Taproot API responses by default, while custom decoders remain available for installer-specific protocols.

The generated Site owns the server integration boundary. Its handlers may initialize Taproot against managed persistence, enforce attribution and `If-Match` revisions, expose search/entity/revision operations, and project data through Diamond for read-only SPARQL. Waystone itself does not import those handlers or assemble a complete Site.

Waystone verifies its adapters against package-owned canonical protocol fixtures and mock behavior plus an exact packed strict consumer. That consumer asserts that TypeScript does not resolve Taproot, Diamond, Comunica, `rdf-parse`, or `lru-cache` declarations through Waystone. A separate conformance consumer checks out Taproot commit `38b97616da07ee349cf30877653acd84d1689139`, builds and packs it, packs Waystone, installs both archives, and proves that their public statement and mutation-command contracts are mutually assignable under strict TypeScript 5.9. The Codex agent creating a Site owns live mutation, stale-conflict, revision, and SPARQL acceptance against that Site's selected artifacts and resources.

Taproot commit `38b97616da07ee349cf30877653acd84d1689139` makes nonblank authored `Statement.text` mandatory for each new statement revision. Waystone's public display model and its local structural Taproot contract require that field. Add and replace operations carry it on the statement; rank, qualifier, and reference commands carry newly authored text explicitly. The editors leave the revision-text control blank when a statement is selected, require JavaScript-nonblank text, and preserve the submitted string byte-for-byte rather than copying the previous revision's text. Removal remains exempt. This is a breaking Taproot contract: Taproot `>=0.3.0 <0.4.0` must be published and qualified before a compatible Waystone release. The release order is Taproot first, then Waystone against the exact published Taproot artifact, then assembled Seedbed acceptance.
